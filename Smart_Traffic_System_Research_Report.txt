================================================================================
2. ABSTRACT
================================================================================

Traffic.AI is a full-stack smart traffic management system that combines
real-time computer vision (YOLOv8), reinforcement learning (PPO via Stable
Baselines3), time-series forecasting (AutoARIMA), and live external API
integration (TomTom Traffic Flow, OpenWeatherMap) to monitor, analyze, and
optimize traffic flow across 11 major Bangalore traffic zones.

The system features a React 18 / TypeScript frontend with 19 pages, a Python
FastAPI backend with 20+ API modules, a Supabase (PostgreSQL) database for
persistent telemetry, and a bi-directional WebSocket architecture for real-time
telemetry streaming at ~10 FPS.

The AI pipeline includes:
  - YOLOv8n for vehicle detection and lane-wise queue counting
  - PPO Reinforcement Learning for adaptive traffic signal control
  - AutoARIMA for 30-minute congestion forecasting
  - RAG (Retrieval Augmented Generation) chatbot with Supabase vector store
  - Automated retraining pipeline for continuous RL model improvement


================================================================================
3. INTRODUCTION
================================================================================

Urban traffic congestion is a critical challenge in Indian metropolitan cities.
Bangalore, one of the fastest-growing tech hubs, experiences severe congestion
at key junctions such as Silk Board, Marathahalli, Hebbal Flyover, KR Puram
Bridge, and Electronic City.

Traffic.AI addresses this problem through a software-only, multi-agent
architecture that:

  1. Monitors real-time traffic conditions using CCTV/RTSP camera feeds
     processed by YOLOv8 computer vision
  2. Optimizes traffic signal phases using a PPO reinforcement learning agent
     trained on a custom 4-way intersection environment
  3. Predicts future congestion using time-series analysis (AutoARIMA) on
     historical telemetry data stored in Supabase
  4. Integrates live external data from TomTom Traffic Flow API and
     OpenWeatherMap API for enhanced situational awareness
  5. Provides a comprehensive web-based command center for traffic operators
     with real-time dashboards, analytics, maps, and PDF report generation

The system is designed specifically for 11 major Bangalore traffic zones with
real GPS coordinates and supports both live API data and a calibrated baseline
fallback system when APIs return 403 errors.

6. AI MODELS USED
================================================================================

6.1 YOLOv8n — Vehicle Detection & Lane Queue Counting
------------------------------------------------------
  File:        backend/vision.py, backend/vision_runtime.py
  Model File:  backend/yolov8n.pt (6.5 MB, pre-trained nano model)
  Library:     Ultralytics
  Input:       CCTV/RTSP video frames (webcam, video file, or stream URL)
  Output:      - Vehicle count per frame
               - Density percentage (count / max_capacity * 100)
               - Lane-wise queue counts (north, south, east, west)
               - Emergency vehicle detection (truck class_id=7)
               - Annotated JPEG frame for live streaming
  Role:        Real-time vehicle detection and spatial queue analysis.
               Classifies vehicles in classes [2:car, 3:motorcycle, 5:bus, 7:truck].
               Runs at ~15 FPS with confidence threshold 0.3.
               Supports hot-reload of custom fine-tuned weights (yolov8n_custom.pt).
  Multi-Camera: Managed by MultiCameraVisionManager class supporting 4 cameras:
               cam-silk-board, cam-marathahalli, cam-hebbal, cam-ecity

6.2 PPO (Proximal Policy Optimization) — Signal Phase Control
--------------------------------------------------------------
  File:        backend/rl_model.py
  Model File:  backend/ppo_traffic_model.zip (139 KB, pre-trained)
  Library:     Stable Baselines3
  Environment: TrafficIntersectionEnv (custom Gymnasium env)
    - Action Space: Discrete(2) — 0=NS_GREEN, 1=EW_GREEN
    - Observation Space: Box(0, 100, shape=(4,)) — [North, South, East, West] queue lengths
    - Reward: Negative sum of queue lengths + variance penalty (fairness)
    - Max Steps: 200 per episode
  Input:       Lane queue counts [N, S, E, W] from YOLO vision or TomTom fallback
  Output:      Optimal signal phase (0=NS_GREEN or 1=EW_GREEN)
  Role:        Adaptive traffic signal control. Decides which direction gets
               green light based on real-time queue observations.
               Falls back to NS_GREEN (action=0) if model is not loaded.
  Training:    train_ppo_model() function with configurable timesteps.
               Default training: 5,000 timesteps.
               MlpPolicy (Multi-Layer Perceptron) neural network.

6.3 PPO Retraining on Historical Data
---------------------------------------
  File:        backend/retrain.py
  Environment: HistoricalTrafficEnv (custom Gymnasium env replaying DB records)
    - Observation Space: Box(0, 500, shape=(4,))
    - Replays actual traffic data from Supabase traffic_data table
  Input:       Historical lane counts (north, south, east, west) from database
  Output:      Updated ppo_traffic_model.zip weights
  Role:        Continuous learning pipeline. Loads historical trajectories
               from Supabase and retrains the PPO model (10,000 timesteps).
               Supports both fine-tuning (loading existing weights) and cold start.

6.4 AutoARIMA — Traffic Congestion Forecasting
------------------------------------------------
  File:        backend/forecasting.py
  Library:     pmdarima
  Input:       Historical density series from Supabase traffic_data table
               (last 100 records for intersection BLR-CORE-1)
  Output:      5-step ahead congestion predictions (list of density values)
  Role:        Time-series forecasting model. Uses auto_arima with stepwise
               selection to find optimal ARIMA parameters. Requires minimum 10
               data points. Logs predictions to signal_logs table.

6.5 Trend Projection — Real-Time Forecasting (prediction_api.py)
-----------------------------------------------------------------
  File:        backend/prediction_api.py
  Method:      _forecast_congestion()
  Input:       Current congestion + recent density history from intersection_snapshots
  Output:      5-minute interval predictions for up to 120 minutes ahead
  Role:        Lightweight linear trend projection using last 2 data points
               for real-time dashboard predictions. Bounded by observed
               historical floor/ceiling. Different from AutoARIMA module.

6.6 YOLOv8 Fine-Tuning Pipeline
----------------------------------
  File:        backend/fine_tune_yolov8.py
  Input:       Indian traffic dataset (stub configuration with 8 classes:
               pedestrian, bicycle, car, motorcycle, auto_rickshaw, bus, tempo, truck)
  Output:      Fine-tuned weights at yolov8n_custom.pt
  Role:        Training pipeline for customizing YOLOv8 on Indian traffic vehicles.
               Currently uses stub dataset.yaml (real dataset not included).
               Supports hot-reload deployment — vision tracker auto-detects new weights.

6.7 RAG Agent — AI Chatbot
----------------------------
  File:        backend/agent.py
  Libraries:   LangChain Community, Supabase Vector Store
  Input:       User query + live weather/traffic telemetry + vector store context
  Output:      Contextual traffic advisory response
  Role:        Retrieval Augmented Generation chatbot. Uses SupabaseVectorStore
               with MockEmbeddings (1536-dim zero vectors) as fallback when
               OpenAI embeddings are unavailable. Falls back to hardcoded
               historical Bangalore traffic data if vector search fails.
               Auto-ingests live weather and traffic alerts hourly.

9. WORKING MECHANISM
================================================================================

9.1 STARTUP SEQUENCE
---------------------
  1. FastAPI app initializes (main.py)
  2. Supabase client created with service role key
  3. MultiCameraVisionManager starts all configured cameras (vision.py)
  4. PPO RL model loaded from ppo_traffic_model.zip (rl_model.py)
  5. Immediate TomTom API poll for all 9 grid nodes
  6. Boot logs seeded into signal_logs table
  7. Background tasks started: TomTom polling (60s), RAG auto-ingest (1h)

9.2 REAL-TIME TELEMETRY LOOP (WebSocket)
------------------------------------------
  Every ~100ms (10 FPS):
  1. Get vision snapshot from primary camera (YOLOv8)
  2. Extract vehicle count, density, lane queues
  3. If vision inactive, fallback to TomTom grid congestion
  4. PPO RL model predicts optimal signal phase from lane queues
  5. Check operator overrides (emergency, AI paused, manual phase)
  6. Collect hardware metrics (CPU, memory via psutil)
  7. Build telemetry payload and send via WebSocket
  8. Every 5s: persist system_metrics + intersection_snapshots to Supabase
  9. Every 30s: insert traffic_data row with weather correlation

9.3 SIGNAL CONTROL LOGIC
--------------------------
  Priority Order:
  1. If AI is paused → use manual_phase or default NS_GREEN
  2. If emergency active → force green on dominant lane direction
  3. Otherwise → PPO RL model prediction (NS_GREEN or EW_GREEN)

9.4 COMMAND CENTER INTERACTION
-------------------------------
  - WebSocket supports bi-directional communication
  - Operator can send commands: emergency toggle, AI pause, forced signal phase
  - REST fallback: POST /api/command/control for same overrides
  - All operator overrides logged to signal_logs in Supabase

9.5 PREDICTION WORKFLOW
-------------------------
  1. GET /api/predict called with current_congestion
  2. Fetch recent density history from intersection_snapshots
  3. Calculate trend from last 2 data points
  4. Project forward at 5-minute intervals up to 120 minutes
  5. Apply per-junction trend delta to all zone snapshots
  6. Generate AI recommendation based on peak predicted congestion

11. LIMITATIONS (BASED ON CODE ANALYSIS)
================================================================================

11.1 YOLOv8 Fine-Tuning Dataset Not Included
  - fine_tune_yolov8.py creates a STUB dataset.yaml
  - No actual labeled Indian traffic images are in the repository
  - The fine-tuning pipeline will catch exceptions and simulate weight deployment

11.2 RAG Agent Uses MockEmbeddings
  - agent.py uses MockEmbeddings (all-zeros 1536-dim vectors) as fallback
  - Real OpenAI embeddings require OPENAI_API_KEY to be valid
  - Vector similarity search with mock embeddings returns arbitrary results

11.3 Emergency Corridor Uses Simple Routing
  - prediction_api.py emergency corridor uses simple intermediate junction selection
  - Comment in code: "in real system: Dijkstra on road graph"
  - No actual road graph or shortest-path algorithm implemented

11.4 TomTom API 403 Fallback
  - TomTom API frequently returns 403 errors (key restrictions)
  - System falls back to get_traffic_baseline() / get_bangalore_baseline()
  - Baseline uses deterministic time-based patterns, not real traffic data

11.5 Camera Sources Not Configured by Default
  - All 4 camera sources depend on environment variables (CCTV_RTSP_URL_*)
  - Without configured RTSP URLs, vision system enters "disconnected" state
  - sample_traffic.mp4 available for testing but not auto-configured

11.6 Auto-Scheduler is a Simple Loop
  - auto_scheduler.py uses time.sleep(86400) instead of proper job scheduling
  - Comment notes: "In Enterprise, replaced by Airflow DAGs or Kubernetes CronJobs"

11.7 Daily Report Email is Simulated
  - auto_scheduler.py send_daily_report_email() only prints to console
  - No actual email sending (smtplib/SendGrid) is implemented

11.8 Anomaly Health Monitor is Simulated
  - auto_scheduler.py anomaly_health_monitor() only prints nominal status
  - No actual DB queries for anomaly detection in the scheduler

11.9 Historical Traffic Data Endpoints Return None
  - traffic_api.py get_historical_traffic_data() returns all None values
  - Source marked as "UNAVAILABLE" — no historical dataset bundled

11.10 Emergency Vehicle Detection Heuristic
  - vision.py detects emergency vehicles by class_id == 7 (truck)
  - This is a proxy heuristic — no dedicated ambulance/fire-truck class trained

11.11 OpenAI API Key Exposure Risk
  - Backend .env contains actual API keys (OpenAI, TomTom, Supabase, OpenWeatherMap)
  - These are committed to the repository and should be rotated

12. CONCLUSION
================================================================================

Traffic.AI demonstrates a comprehensive, production-oriented smart traffic
management system architecture for the Bangalore metropolitan area. The project
successfully integrates:

  - Real-time computer vision (YOLOv8) for vehicle detection and counting
  - Reinforcement learning (PPO) for adaptive signal phase optimization
  - Time-series forecasting (AutoARIMA + trend projection) for congestion prediction
  - External API integration (TomTom + OpenWeatherMap) for live traffic and weather data
  - A full-stack web application (React + FastAPI + Supabase) with 19 pages and 20+ API endpoints
  - Real-time bi-directional WebSocket telemetry at 10 FPS
  - Comprehensive operator tools including command center, audit logs, and PDF reports

The system covers 11 major Bangalore traffic zones with real GPS coordinates and
provides both live API data and calibrated baseline fallbacks. The architecture
supports multi-camera management, hot-reload of model weights, continuous RL
retraining, and bulk data ingestion.

Key limitations include the absence of a labeled Indian traffic dataset for YOLO
fine-tuning, mock embeddings in the RAG agent, simplified emergency routing logic,
and simulated scheduler functions. The API keys in the repository should be
rotated for production deployment.

The project represents a strong foundation for a real-world smart traffic system
that combines multiple AI techniques with modern web technologies and cloud
database services.
